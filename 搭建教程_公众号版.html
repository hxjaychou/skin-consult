皮肤健康咨询网站搭建教程

从零开始，手把手教你搭建一个专业的皮肤健康咨询网站

本教程基于真实搭建经历编写，包含所有踩过的坑和解决方法

一、你将得到什么

一个专业的皮肤健康咨询网站，包含：

功能说明
首页品牌展示，引导客户开始咨询
在线问诊客户填写症状表单，数据自动存入云端数据库
产品展示治疗类 + 护肤类产品卡片，客户自助了解
管理后台查看所有咨询记录，更新跟进状态，填写专业分析
全终端适配手机、平板、电脑、暗色模式自动适应

💡 技术栈：纯 HTML + Supabase（云端数据库）+ Cloudflare Pages（免费托管）

部署后地址格式：https://你的项目名.pages.dev

二、准备工作

你需要提前准备：

  1. 一个 GitHub 账号（免费注册）
  2. 一个 Cloudflare 账号（免费注册）
  3. 你的 Supabase 项目信息（URL 和 Key）
  4. 网站源代码文件（index.html 和 setup-supabase.sql）

三、完整搭建步骤

第一步：创建 Supabase 数据库

Supabase 是一个免费的云端数据库，我们用它来存储客户的咨询记录。

1.1 注册并登录

  • 打开 https://supabase.com
  • 点击右上角 Sign In
  • 选择 Sign in with GitHub（用你的 GitHub 账号登录）

💡 踩坑记录 #1： 我第一次用邮箱注册，后来发现用 GitHub 登录更方便，因为后面 Cloudflare 也要用 GitHub 授权。建议一开始就用 GitHub 账号登录所有服务。

1.2 创建新项目

  • 登录后，点击 New Project
  • 填写信息：
    • Organization：选你的组织，或点 Create a new organization
    • Project name：填 skin-consult
    • Database Password：设一个复杂密码，一定要记住！
    • Region：选 Southeast Asia (Singapore) 或 Northeast Asia (Tokyo)（离中国最近）
  • 点击 Create new project
  • 等待项目初始化完成（通常1-2分钟，状态变绿）

⚠️ 踩坑记录 #2（重要）： 区域选择很重要！我一开始选了美国区域，结果访问速度很慢。后来改成新加坡区域，速度提升明显。如果你的客户主要在中国，一定要选亚洲区域。

1.3 创建数据库表

  • 左侧菜单点 SQL Editor
  • 点 New query
  • 把 setup-supabase.sql 文件的全部内容复制粘贴进去
  • 点右下角 Run 按钮
  • 看到 Success 表示成功

📌 这个脚本做了什么？
• 创建 consultations 表（存储咨询记录，13个字段）
• 开启行级安全策略（RLS）
• 允许匿名用户读写（前端直连需要）
• 创建索引加速查询
• 创建照片存储桶（如果需要上传照片）

🚨 踩坑记录 #3（大坑！）： 我第一次建表时，脚本只创建了5个字段（id, created_at, nickname, contact, category），但网站需要13个字段。结果提交表单时一直报错，以为是权限问题，查了半天才发现是表结构不对。

解决方法： 删除旧表重新创建。在 SQL Editor 中执行：

DROP TABLE IF EXISTS consultations;

然后重新运行完整的建表脚本。

💡 教训： 建表前一定要检查脚本是否完整，最好先在本地用文本编辑器确认字段数量。

1.4 获取 API 信息

  • 左侧菜单点 Settings → API
  • 找到以下两个值并复制保存：
名称说明格式示例
Project URL数据库地址https://xxxxxxxx.supabase.co
anon public Key访问密钥以 eyJ 开头的长字符串

⚠️ 重要： 这两个值后面要填入网站代码中，不要泄露给别人。

第二步：准备网站代码

2.1 你需要的文件

文件作用
index.html网站主页面（单页应用，包含所有功能）
setup-supabase.sql数据库建表脚本（第一步已用）

2.2 填入你的数据库配置

用文本编辑器打开 index.html，找到第 415-416 行附近：

const URL = 'https://pyujfsvhtpgxykvhvnse.supabase.co';
const KEY = 'sb_publishable_FxeCK7sLnE2H3RK_hGEA6A_ZEgmZttA';

替换为你自己的 Supabase 信息：

const URL = '你的Project URL';
const KEY = '你的anon public Key';

同时修改管理后台密码（第 417 行）：

const DB = '8866';  // 改成你自己的密码

保存文件。

⚠️ 踩坑记录 #4： 我一开始把 Key 填错了位置，把 Project URL 填到了 Key 的位置。结果提交表单时一直报 401 错误，以为是权限问题。后来仔细检查才发现是填反了。

解决方法： 填完后，用文本编辑器的搜索功能检查：

  • 搜索 supabase.co 确认 URL 格式正确
  • 搜索 eyJ 确认 Key 以这个开头

2.3 本地测试

  • 双击打开 index.html，或用浏览器打开
  • 测试以下功能：
    • ✅ 首页显示正常
    • ✅ 点「开始咨询」跳转到问诊表单
    • ✅ 填写表单并提交，显示「提交成功」
    • ✅ 点「产品」查看产品列表
    • ✅ 点「管理」输入密码登录后台
    • ✅ 后台能看到刚才提交的咨询记录

💡 踩坑记录 #5： 本地测试时，我发现表单提交后没有反应，控制台报错说找不到 consultations 表。检查后发现是 Supabase 项目还没初始化完成，SQL 脚本还没执行。

解决方法： 确保 Supabase 项目状态是 Active，SQL 脚本已执行成功，再测试。

第三步：上传到 GitHub

3.1 创建新仓库

  • 打开 https://github.com
  • 点右上角 + → New repository
  • 填写：
    • Repository name：填 skin-consult
    • Description：填 皮肤健康咨询网站（可选）
    • 选 Public（Cloudflare Pages 需要公开仓库，或付费私有）
    • 不要勾选 "Add a README file"（保持空仓库）
  • 点 Create repository

3.2 上传文件

  • 在新仓库页面，点击 Add file → Upload files
  • 把 index.html 文件拖到灰色虚线框区域
  • ⚠️ 重要：勾选 "Replace all files in this commit"（如果有这个选项）
  • 点绿色的 Commit changes 按钮

🚨 踩坑记录 #6（最崩溃的坑！）： 我第一次上传时，没有看到 "Replace all files in this commit" 选项，以为同名文件会自动替换。结果上传后，GitHub 上显示的是旧文件，新文件没生效。

表现： 网站打开后还是旧版本，没有产品页，没有响应式适配。

排查过程：

  1. 检查 GitHub 仓库，发现文件确实是旧的
  2. 尝试用 GitHub API 更新，但需要认证
  3. 尝试用 git push，但终端没有 GitHub 认证
  4. 最后发现是上传时没有正确替换文件

解决方法：

  1. 打开 https://github.com/你的用户名/仓库名/edit/main/index.html
  2. 全选删除旧内容（Cmd+A → Delete）
  3. 打开本地文件，全选复制（Cmd+A → Cmd+C）
  4. 粘贴到 GitHub 编辑器里
  5. 点 Commit changes

💡 教训： 上传文件时，一定要确认 "Replace all files in this commit" 被勾选。如果没看到这个选项，就用编辑器方式替换，更可靠。

✅ 踩坑记录 #7： 上传后等了很久，Cloudflare 还是显示旧版本。后来发现是浏览器缓存问题。

解决方法： 强制刷新浏览器（Ctrl+Shift+R 或 Cmd+Shift+R），或者清除浏览器缓存。

第四步：部署到 Cloudflare Pages

4.1 注册并登录 Cloudflare

  • 打开 https://dash.cloudflare.com
  • 点 Sign Up 注册（可以用邮箱或 GitHub 账号）
  • 登录后进入控制台

4.2 创建 Pages 项目

  • 左侧菜单点 Workers & Pages
  • 点 Create application
  • 选 Pages 标签
  • 点 Connect to Git

4.3 连接 GitHub 仓库

  • 选择 GitHub
  • 如果是第一次，需要授权 Cloudflare 访问你的 GitHub：
    • 点 Authorize Cloudflare Workers
    • 登录 GitHub 确认授权
  • 在仓库列表中找到并选 skin-consult
  • 点 Begin setup

💡 踩坑记录 #8： 我第一次授权时，GitHub 提示需要两步验证，但我的手机没在身边，卡了好久。

解决方法： 提前确保手机在身边，或者用电脑上的 GitHub Desktop 登录，避免两步验证的麻烦。

4.4 配置部署选项

  • Project name：填 skin-consult（或你喜欢的名字）
  • Production branch：选 main
  • Build settings：全部留空（纯静态站点不需要构建命令）
  • Build output directory：填 /（根目录）
  • 点 Save and Deploy

⚠️ 踩坑记录 #9： 我一开始在 Build command 里填了 npm install，结果部署失败，提示找不到 package.json。

解决方法： 纯 HTML 网站不需要构建命令，全部留空就行。Cloudflare 会自动检测到这是一个静态站点。

4.5 等待部署完成

  • 等待1-2分钟，状态变成 Success
  • 你会得到一个免费域名：https://skin-consult.pages.dev
  • 点击这个链接就能访问你的网站了！

✅ 踩坑记录 #10： 部署完成后，我打开网站发现是空白页。检查 Cloudflare 日志，发现是文件路径错误。

解决方法： 确保 index.html 在仓库的根目录，而不是在子文件夹里。如果是子文件夹，需要在 Build output directory 里填正确的路径。

五、使用指南

客户端（客户使用）

  1. 打开网站链接
  2. 点「开始咨询」
  3. 填写昵称、微信号、问题类型、症状描述
  4. 点「提交咨询」
  5. 等待回复

管理端（你自己使用）

  1. 打开网站链接
  2. 点导航栏的「管理」
  3. 输入密码（默认 8866，可自行修改）
  4. 查看所有咨询记录
  5. 点击某条记录查看详情
  6. 更新状态（待回复 → 已回复 → 跟进中 → 已完成）
  7. 填写「专业分析」给客户建议

六、后续更新

每次修改代码后：

  1. 打开 GitHub 仓库
  2. 点 Add file → Upload files
  3. 上传新的 index.html（同名文件会自动替换）
  4. ⚠️ 确保勾选 "Replace all files in this commit"
  5. 点 Commit changes
  6. Cloudflare 会自动检测并重新部署（1-2分钟）

💡 最佳实践： 每次更新前，先在本地测试。更新后，强制刷新浏览器检查效果。

七、常见问题

Q：网站打不开？

  • 检查 Cloudflare 部署状态是否为 Success
  • 尝试清除浏览器缓存（Ctrl+Shift+R 强制刷新）
  • 检查域名是否正确

Q：提交表单后没有反应？

  • 检查 Supabase 的 URL 和 Key 是否正确
  • 打开浏览器开发者工具（F12）查看 Console 有没有报错
  • 确认 Supabase 项目状态是 Active

Q：管理后台登录不了？

  • 确认密码是否正确（默认 8866）
  • 检查代码中的密码常量 DB 是否被修改

Q：手机端显示不正常？

  • 网站已做全终端适配，正常情况下会自动适应
  • 如果还有问题，尝试清除缓存或换浏览器

Q：如何修改网站内容？

  • 用文本编辑器打开 index.html
  • 修改对应的文字、颜色、布局
  • 重新上传到 GitHub，Cloudflare 会自动更新

十、技术支持

如果遇到问题：

  1. 检查浏览器控制台：F12 → Console 查看错误信息
  2. 检查网络请求：F12 → Network 查看 API 调用是否成功
  3. 查看 Supabase 日志：Supabase 控制台 → Logs
  4. 查看 Cloudflare 日志：Cloudflare Pages → Deployments → 查看构建日志

常见错误代码：

  • 401：API Key 错误（检查 URL 和 Key 是否正确）
  • 403：权限不足（RLS 策略问题，检查 SQL 脚本）
  • 404：表不存在（需要运行建表脚本）
  • 500：服务器错误（检查 Supabase 状态）

十一、我的经验总结

1. 先小后大

先做一个最小可用版本，测试通过后再逐步添加功能。我一开始就想做全部功能，结果到处出错。

2. 每步验证

每完成一个步骤，都要测试一下。比如：

  • 建表后，先测试能否插入数据
  • 上传代码后，先本地测试
  • 部署后，先检查基本功能

3. 善用浏览器工具

F12 开发者工具是你的好朋友：

  • Console 看错误信息
  • Network 看网络请求
  • Elements 看页面结构

4. 缓存是万恶之源

很多时候问题不是代码错了，而是浏览器缓存了旧版本。养成强制刷新的习惯。

5. 版本控制很重要

每次修改前，先备份当前版本。我好几次改坏了又没备份，只能重头再来。

6. 不要怕问

遇到问题先搜索，搜索不到就问。AI 工具可以帮你快速定位和解决问题。
